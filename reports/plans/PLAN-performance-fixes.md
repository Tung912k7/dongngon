# PLAN: Next.js Performance & Core Web Vitals Optimization

## Goal
Resolve 4 key performance errors detailed in `bug.md` by targeting render-blocking architecture, cutting unused Javascript, and deferring legacy/heavy payload loading.

## Steps
1. **Optimize Heavy Global Components (`app/layout.tsx`)**
   - The Root Layout loads `components/GuideNotification.tsx` synchronously (shipping `framer-motion` to EVERY first-load packet).
   - *Fix*: Replace the static import with `next/dynamic` using `{ ssr: false }`.
   - The Root Layout also loads `components/SmoothScroll.tsx` synchronously (shipping `lenis`). 
   - *Fix*: Replace the static import with `next/dynamic` using `{ ssr: false }`.

2. **Optimize Telemetry/Analytics Loading (`app/providers.tsx`)**
   - The file imports `posthog-js` locally globally. Because it’s an active chunk evaluated synchronously, it blocks paint unnecessarily (wasting ~150kb).
   - *Fix*: Instead of executing `posthog.init` statically on file load, wrap the init logic inside the Client Component `useEffect` block so the code only parses after React mounts, effectively stopping it from becoming a render-blocker for First Contentful Paint.

3. **Validate Configs**
   - Verify that there are no old redundant global CSS scripts triggering network depth warnings.
   - Clean up code spacing.

## Exit Criteria
- Output JS bundles generated by Webpack are cleanly detached.
- `framer-motion` and `lenis` chunks are correctly separated and lazy-loaded via `next/dynamic`.
- `posthog-js` execution is delayed until after React paints.
